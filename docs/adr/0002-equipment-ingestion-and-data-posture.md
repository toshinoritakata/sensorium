# 機材データ取込の3アダプタ統一と、出典付き・非ミラーの法務姿勢

工業用センサー（フォト/近接/測距/カーテン/感圧/PIR/レーダー）まで機材DBを広げるにあたり、取込を3アダプタ（`api-distributor`=DigiKey主・Mouser補、`flat-file`=CSV/Misumi/RS、`url-extract`=任意URL）に統一し、いずれも共通パイプライン「**候補(unverified) → 人レビュー → 有効**」を通す。カタログ項目→自社 envelope のマッピングは Claude 抽出＋人レビューで行う。

ライブ照会でなく取込＋自社DB保持を選ぶのは、What-if の決定論再計算（ADR-0001）がオフラインかつ即時であるべきで、毎回の外部API照会と両立しないため。一方で自動取込品を即有効化せず候補ゲートを挟むのは、誤マッピング・欠損値がそのまま判定に入ると成立条件の信頼が崩れるから（confidence/provenance と同じ流儀）。

法務姿勢は **出典付き・on-demand・非ミラー**: 仕様は出典リンク付きで参照保持し、ディストリビュータのカタログをバルク複製しない。各 API の User Agreement（DigiKey の保存/レート規約、Mouser の「自アプリ内表示」限定ライセンス等）と第三者ページの ToS を実装前に確認する。MonotaRO のように公開APIが無く調達寄りのソースは、バルク取得でなく `url-extract` で1点ずつ on-demand 取込する。

この姿勢は freshness と全自動化を一部犠牲にするが、法的安全性と判定の正しさを優先する。後から「全自動ライブ同期」に倒すと、オフライン性・再現性・ToS 安全性が崩れるため明文化して固定する。
